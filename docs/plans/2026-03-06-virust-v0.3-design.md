# Virust v0.3 Design

**Goal:** Complete v0.2's unfinished features to make Virust work with real frontends

**Approach:** Incremental Integration - build and validate each component before wiring them together

**Status:** ✅ **Completed** - 2026-03-06

**Release Notes:** See [v0.3 Release Notes](./2026-03-06-virust-v0.3-release.md)

---

## Architecture Overview

Virust v0.3 completes the filesystem-based routing system started in v0.2. The framework uses a **single-port architecture** where the Rust backend serves everything on port 3000:

- **Static files** (web/) → served directly from `:3000/`
- **API routes** (api/) → served from `:3000/api/*`
- **WebSocket** → served from `:3000/ws`
- **TypeScript types** → generated at `:3000/api/__types`

This is similar to Next.js App Router - developers work with a single port and the framework handles the rest.

### Key Principles

1. **Filesystem Routing**: `api/chat/route.rs` → `/api/chat`, `api/todos/[id]/route.rs` → `/api/todos/:id`
2. **Type Safety**: Rust structs generate TypeScript interfaces automatically
3. **Developer Experience**: One command (`virust dev`) runs everything with hot reload
4. **No Build Step**: Frontend changes reload instantly via WebSocket HMR

## Components

### Phase 1: Parameter Extraction

**Goal:** Make `#[path]` and `#[body]` attributes actually work.

Currently, route handlers receive raw `String` parameters:

```rust
#[post]
async fn create_todo(payload: String) -> String {
    let input: CreateTodoRequest = serde_json::from_str(&payload)?;
    // ...
}
```

After v0.3, they receive typed parameters:

```rust
#[post]
async fn create_todo(#[body] payload: CreateTodoRequest) -> String {
    // payload is already deserialized
    // ...
}

#[get]
async fn get_todo(#[path] id: String) -> String {
    // id is extracted from URL path
    // ...
}
```

**Implementation:**
- Extend `virust-macros` to generate extractor code
- Path parameters: Use Axum's `Path<T>` extractor
- Body parameters: Use Axum's `Json<T>` extractor
- Support multiple parameters: `#[path] id: String, #[body] data: Request`

**Benefits:**
- Less boilerplate code
- Compile-time type checking
- Better error messages (Axum handles validation)

### Phase 2: Complete TypeScript Generation

**Goal:** Generate complete TypeScript interfaces with field definitions.

Currently, the TypeScript generator creates empty interface stubs:

```typescript
export interface CreateTodoRequest {
  // Field definitions will be added in future versions
}
```

After v0.3, it generates complete interfaces:

```typescript
export interface CreateTodoRequest {
  title: string;
  description?: string;
}

export interface TodoResponse {
  id: string;
  title: string;
  description?: string | null;
  completed: boolean;
  created_at: number;
}
```

**Implementation:**
- Use `syn` crate to parse Rust struct definitions
- Map Rust types to TypeScript types:
  - `String` → `string`
  - `i32/i64` → `number`
  - `bool` → `boolean`
  - `Option<T>` → `T | null`
  - `Vec<T>` → `T[]`
- Extract field names and visibility from struct AST
- Generate TypeScript declarations in `typescript.rs`

**Benefits:**
- Frontend gets real type safety
- No need to manually duplicate types
- Catches API mismatches at compile time

### Phase 3: Route Discovery Integration

**Goal:** Wire up the `api/` directory scanning so routes auto-register.

Currently, route registration is manual and disconnected from the filesystem:

```rust
// In api/chat/route.rs - route registers to inventory
#[ws]
async fn chat(msg: ChatMessage) -> ChatResponse { ... }

// But runtime doesn't know about api/chat/route.rs
```

After v0.3, the runtime automatically discovers and registers routes:

```rust
// At startup, runtime:
// 1. Scans api/ directory
// 2. Loads each route.rs module
// 3. Collects inventory-registered routes
// 4. Builds Axum router
```

**Implementation:**
- Build on existing `inventory` pattern in `virust-macros`
- Create `build_router()` function that:
  - Walks `api/` directory tree
  - Forces module loading for each `route.rs`
  - Reads from `inventory` registry
  - Constructs Axum route tree
- Handle dynamic routes: `api/todos/[id]/route.rs` → `/api/todos/:id`

**Benefits:**
- No manual route registration
- Filesystem maps 1:1 to URL structure
- Adding routes is just creating files

### Phase 4: End-to-End Single-Port Server

**Goal:** Make `virust dev` serve everything on one port with HMR.

Currently, templates are set up for Vite proxy on port 5173 → 3000.

After v0.3, development mode:

```bash
virust dev
# 🚀 Server running on http://127.0.0.1:3000
# ✨ Frontend: /web/index.html
# 📡 API: /api/*
# 🔌 WebSocket: /ws
# 🔄 HMR enabled for web/
```

**Implementation:**
- Add static file serving to Axum router
- Serve `web/` directory at root path
- WebSocket endpoint for HMR:
  - Watch `web/` files for changes
  - Broadcast reload message to connected clients
  - Client-side JS handles hot reload
- Update `virust-cli/src/dev.rs` to run single server
- Update templates to remove Vite proxy config

**Frontend HMR client:**
```javascript
// Auto-injected by virust dev
const ws = new WebSocket('ws://localhost:3000/ws');
ws.onmessage = (msg) => {
  if (msg.data === 'reload') location.reload();
};
```

**Benefits:**
- True single-command development
- Instant feedback on frontend changes
- Matches Next.js developer experience

## Data Flow

### Request Lifecycle

```
1. Browser → http://localhost:3000/api/todos
2. Axum router matches route
3. Path extractor extracts :id if present
4. Body extractor parses JSON to struct
5. Route handler executes with typed params
6. Response serialized to JSON
7. Returned to browser
```

### Type Generation Flow

```
1. cargo build parses route.rs files
2. virust-macros extracts type info from functions
3. Types stored in global inventory registry
4. At runtime, TypeScriptGenerator reads registry
5. Generates complete interfaces with fields
6. Served at /api/__types endpoint
```

### Development Flow

```
1. User runs: virust dev
2. Server starts on :3000
3. File watcher monitors web/ and api/
4. Frontend changes → WebSocket reload
5. Backend changes → auto-recompile + restart
6. Frontend fetches /api/__types for updated types
```

## Error Handling

### Parameter Extraction Errors

**Path parameter mismatch:**
- Bad URL pattern → 404 (Axum handles)
- Type conversion failure (e.g., `id: i32` receives "abc") → 400 with error message

**Body parsing errors:**
- Invalid JSON → 400 with parsing error details
- Missing required fields → 400 with field validation errors
- Type mismatch (e.g., expected `bool` got string) → 400

All extraction errors return structured JSON:

```json
{
  "error": "Invalid request",
  "details": "Failed to parse 'completed': expected boolean, got string"
}
```

### Route Discovery Errors

**Compilation errors:**
- Syntax error in route.rs → Rust compiler error at build time
- Missing module → `cargo build` fails with clear message

**Runtime errors:**
- Conflicting routes (same URL, different methods) → panic on startup with conflict details
- Invalid route path → warning logged, route skipped

### TypeScript Generation Errors

**Parse errors:**
- Complex generic types → log warning, use `any` for that type
- Recursive types → log warning, use `any` for recursive field
- Unknown attributes → ignored, not included in types

**Never crash:** TypeScript generation errors don't crash server, just log and serve partial types

## Testing Strategy

### Unit Tests

**Parameter Extraction:**
- Test PathExtractor with various URL patterns
- Test JsonExtractor with valid/invalid JSON
- Test type conversion (string → i32, string → bool, etc.)
- Test error messages for extraction failures

**TypeScript Generation:**
- Test struct field parsing
- Test type mapping (Rust → TypeScript)
- Test Option<T> → T | null
- Test Vec<T> → T[]
- Test nested structs

**Route Discovery:**
- Test scanning api/ directory
- Test route registration from inventory
- Test dynamic route handling (`[id]` → `:id`)
- Test conflicting route detection

### Integration Tests

**End-to-end API calls:**
- Create test project with todo template
- Test GET /api/todos returns empty array
- Test POST /api/todos with valid body creates todo
- Test POST /api/todos with invalid JSON returns 400
- Test GET /api/todos/:id returns specific todo
- Test PUT /api/todos/:id updates todo
- Test DELETE /api/todos/:id deletes todo

**TypeScript generation:**
- Fetch /api/__types endpoint
- Verify generated types match Rust structs
- Verify all routes have client functions
- Verify field types are correct

### Manual Testing

**virust dev workflow:**
1. Create new project: `virust init test-app -t todo`
2. Run dev server: `virust dev`
3. Open http://localhost:3000
4. Create todo via UI
5. Edit web/main.js
6. Verify page reloads automatically
7. Edit api/todos/route.rs
8. Verify server recompiles and restarts
9. Verify todos still work after changes

## Success Criteria

v0.3 is complete when:

1. **Parameter Extraction Works:**
   - [ ] `#[path] id: String` extracts URL parameters
   - [ ] `#[body] data: MyStruct` parses JSON to structs
   - [ ] Multiple parameters work: `#[path] id: String, #[body] data: Request`
   - [ ] Extraction errors return 400 with clear messages

2. **TypeScript Generation Complete:**
   - [ ] All struct fields are extracted
   - [ ] Rust types map correctly to TypeScript types
   - [ ] Option<T> generates `T | null`
   - [ ] Vec<T> generates `T[]`
   - [ ] /api/__types serves valid TypeScript

3. **Route Discovery Connected:**
   - [ ] Scanning api/ directory finds all routes
   - [ ] Routes auto-register on startup
   - [ ] Dynamic routes (`[id]`) work correctly
   - [ ] No manual route registration needed

4. **Single-Port Development:**
   - [ ] `virust dev` serves on single port (:3000)
   - [ ] Static files served from `/`
   - [ ] API routes work at `/api/*`
   - [ ] WebSocket HMR works for frontend changes
   - [ ] Backend changes trigger recompile + restart

5. **Templates Work:**
   - [ ] `virust init myapp -t chat` creates working chat app
   - [ ] `virust init myapp -t todo` creates working todo app
   - [ ] Both templates use parameter extraction
   - [ ] Both templates have complete TypeScript types
   - [ ] Both templates work with single-port dev mode

6. **No Breaking Changes:**
   - [ ] All v0.2 tests still pass
   - [ ] Existing templates still work (with minor updates)
   - [ ] Filesystem routing unchanged
   - [ ] `#[ws]`, `#[get]`, `#[post]` macros unchanged
