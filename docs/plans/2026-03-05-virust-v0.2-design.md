# Virust v0.2 Design Document

**Date:** 2026-03-05
**Status:** Approved
**Version:** 0.2.0

---

## Overview

Building on v0.1, v0.2 transforms Virust into a true Next.js-inspired full-stack framework with working templates. The focus is on **real-world usability** - making Virust actually work with frontends through filesystem routing, runtime TypeScript generation, parameter extraction, and complete example applications.

**Key Goal:** Developers can run `virust init chat-app`, `cd chat-app`, `virust dev`, and have a working full-stack app with real-time WebSocket communication, persistent data, and type-safe frontend/backend integration.

---

## Architecture

### Full-Stack Development Model

```
Developer runs: virust dev
    │
    ├─► Vite dev server (http://localhost:5173)
    │   └─► Serves frontend from web/
    │   └─► Fetches types from http://localhost:3000/__types
    │   └─► Proxies /api and /ws to backend
    │
    └─► Virust backend (http://localhost:3000)
        ├─► Scans api/**/*.rs at startup
        ├─► Registers all handlers (GET, POST, WS, etc.)
        ├─► Serves /__types endpoint (TypeScript definitions)
        ├─► Handles WebSocket connections (JSON-RPC)
        └─► Handles HTTP requests with parameter extraction
```

### Project Structure (True Next.js Style)

```
myapp/
├ api/
│  ├ chat/
│  │  └ route.rs          → WebSocket handler at /api/chat
│  ├ users/
│  │  ├ [id]/
│  │  │  └ route.rs       → HTTP handlers at /api/users/:id
│  │  └ route.rs          → HTTP handlers at /api/users
├ web/
│  ├ index.html
│  └ src/
│     └ types/
│        └ generated.ts   # Auto-generated from /__types
├ src/
│  └ main.rs              # Binary entry point
└ Cargo.toml
```

**Key Design Decisions:**

1. **`api/` at root level** - Separate from `src/` Rust code, mirrors Next.js structure
2. **`route.rs` files** - Each directory can have one `route.rs` defining handlers
3. **Function-based HTTP handlers** - `pub async function GET()`, `POST()`, etc. matching Next.js
4. **Dynamic segments** - `[id]`, `[slug]` map to `:id`, `:slug` in URL paths
5. **Full-stack dev mode** - Single command runs backend + frontend with proxying

## Components

### RouteRegistry (virust-runtime)

```rust
pub struct RouteRegistry {
    http_routes: HashMap<String, HttpHandler>,
    ws_handlers: HashMap<String, WsHandler>,
    type_definitions: HashMap<String, TypeDefinition>,
}

impl RouteRegistry {
    pub fn register_http(&mut self, path: String, handler: HttpHandler);
    pub fn register_ws(&mut self, path: String, handler: WsHandler);
    pub fn register_type(&mut self, name: String, def: TypeDefinition);
    pub fn generate_typescript(&self) -> String;
}
```

**Purpose:** Central registry collecting all discovered handlers and their type information.

### RouteDiscovery (virust-runtime)

```rust
pub fn discover_routes(api_dir: &Path) -> Result<RouteRegistry> {
    // 1. Walk api/**/*.rs looking for route.rs files
    // 2. Extract path: api/chat/route.rs → /api/chat
    // 3. Extract dynamic segments: api/[id]/route.rs → /api/:id
    // 4. Compile each route.rs as test harness to trigger macros
    // 5. Macros register handlers in RouteRegistry via inventory

```

**Purpose:** Scans `api/` directory and triggers route registration.

### Enhanced Macros (virust-macros)

**HTTP Handler Syntax:**
```rust
// api/todos/route.rs
pub async function GET(
    State(persistence): State<Arc<dyn Persistence>>
) -> Json<Vec<Todo>> {
    let todos = persistence.list().await?;
    Json(todos)
}

pub async function POST(
    #[body] input: TodoInput,
    State(persistence): State<Arc<dyn Persistence>>
) -> Json<Todo> {
    let todo = Todo::new(input.title);
    persistence.create(todo).await?;
    Json(todo)
}
```

**Dynamic Route with Path Parameters:**
```rust
// api/[id]/route.rs
pub async function GET(
    #[path] id: String,
    State(persistence): State<Arc<dyn Persistence>>
) -> Json<Todo> {
    let todo = persistence.get_by_id(&id).await?;
    Json(todo)
}

pub async function PUT(
    #[path] id: String,
    #[body] updates: TodoUpdate,
    State(persistence): State<Arc<dyn Persistence>>
) -> Json<Todo> {
    let todo = persistence.update(&id, updates).await?;
    Json(todo)
}
```

**WebSocket Handler:**
```rust
// api/chat/route.rs
#[ws]
pub async function chat(
    #[body] msg: ChatMessage,
    State(persistence): State<Arc<dyn Persistence>>
) -> ChatResponse {
    ChatResponse { ok: true }
}
```

**Purpose:** Provides Next.js-style developer experience with named functions and automatic route registration.

### TypeScriptEndpoint (virust-runtime)

```rust
pub async fn types_handler(State(registry): Registry) -> Response {
    let ts = registry.generate_typescript();
    Response::builder()
        .header("content-type", "application/typescript; charset=UTF-8")
```

**Purpose:** Runtime endpoint that returns TypeScript definitions for all registered handlers.

## Data Flows

### Application Startup Flow

```
1. Developer runs: virust dev
   │
2. DevOrchestrator starts:
   │
   ├─► Vite dev server (background process)
   │   └─► Serves web/ directory
   │   └─► Runs Vite HMR for frontend
   │
   └─► Virust backend (main process)
       │
       ├─► RouteDiscovery scans api/**/*.rs
       │   └─► For each route.rs file:
       │       ├─► Compiles as test harness
       │       └─► Macros register handlers in RouteRegistry
       │
       ├─► RouteRegistry populated with all routes
       │   ├─► /api/chat → chat handler (ws)
       │   ├─► /api/todos → todos handlers (http)
       │   └─► Type definitions collected
       │
       ├─► HTTP server starts on :3000
       │   ├─► GET /__types → TypeScript definitions
       │   ├─► WebSocket /ws → JSON-RPC handler
       │   └─► API routes → matched from registry
       │
       └─► File watcher active
           └─► On change: rebuild backend, restart
```

### Runtime Request Flow

```
Frontend → http://localhost:5173 (Vite)
    │
    ├─► /api/todos → proxied to → http://localhost:3000/api/todos
    │   └─► RouteRegistry matches → pub async function GET()
    │   └─► Extracts #[path] and #[body] parameters
    │   └─► Returns JSON response
    │
    ├─► /ws → proxied to → ws://localhost:3000/ws
    │   └─► WebSocket upgrade
    │   └─► RouteRegistry matches → #[ws] pub async function chat()
    │   └─► JSON-RPC messages dispatched
    │   └─► Returns JSON-RPC response
    │
    └─► /src/types/generated.ts → fetched on build
        └─► Calls GET /__types
        └─► RouteRegistry.generate_typescript()
        └─► Saves to web/src/types/generated.ts
```

### TypeScript Generation Flow

```
1. Frontend builds (or dev server refreshes)
   │
2. Fetches: GET http://localhost:3000/__types
   │
3. RouteRegistry.generate_typescript():
   │   ├─► For each registered handler:
   │   │   ├─► Extract input types (params, body)
   │   │   └─► Extract output types (return type)
   │   └─► Generate TypeScript interfaces and functions
   │
4. Returns complete TypeScript file:
   export interface ChatMessage { message: string }
   export interface ChatResponse { ok: boolean }
   export function chat(msg: ChatMessage): Promise<ChatResponse>
   │
5. Frontend imports: import { chat } from './types/generated'
```

## Implementation Phases

### Phase 1: Route Discovery & Registration

**Goal:** Implement filesystem-based route discovery scanning `api/` directory.

**Tasks:**

1. Create `RouteRegistry` struct with http/ws/type storage
2. Build `RouteDiscovery` to scan `api/**/*.rs` files
3. Implement test harness compilation to trigger macros
4. Add `inventory` global collection for route registration
5. Extract URL path from file path (api/chat/route.rs → /api/chat)
6. Handle dynamic segments [id], [...slug], etc.
7. Update macros to register with route paths

**Success Criteria:**
- Routes in `api/chat/route.rs` automatically registered
- Routes in `api/users/[id]/route.rs` registered with dynamic params
- No manual route registration required

---

### Phase 2: Parameter Extraction Attributes

**Goal:** Implement `#[path]` and `#[body]` attributes for extracting request data.

**Tasks:**

1. Update proc-macros to parse function names (GET, POST, etc.)
2. Implement function name → HTTP method mapping
3. Implement `#[path("param")]` attribute to extract from URL
4. Implement `#[body]` attribute to parse JSON body
5. Add parameter extraction middleware in runtime
6. Support multiple `#[path]` attributes for complex routes
7. Write tests for parameter extraction

**Success Criteria:**
- `pub async function GET()` handles GET requests
- `#[path("id")]` extracts `:id` from URL
- `#[body]` parses JSON to structs
- Multiple handlers per file supported

---

### Phase 3: Runtime TypeScript Generation

**Goal:** Expose `/__types` endpoint that generates TypeScript from registered handlers.

**Tasks:**

1. Extend `RouteRegistry` to collect type information
2. Create `TypeGenerator` to convert Rust types to TypeScript
3. Implement `/__types` HTTP endpoint
4. Generate interfaces for all request/response types
5. Generate typed client functions (chat(), getTodos(), etc.)
6. Update macros to register type definitions
7. Add build script for templates to fetch types

**Success Criteria:**
- GET /__types returns valid TypeScript
- All handler types are exported
- Frontend can import and use types
- Types stay in sync with Rust code

---

### Phase 4: Full-Stack Dev Mode

**Goal:** `virust dev` orchestrates backend + Vite with proxying.

**Tasks:**

1. Implement `DevOrchestrator` in CLI
2. Spawn Vite dev server on :5173 (background)
3. Start Virust backend on :3000 (foreground)
4. Create Vite config in `web/` directory
5. Configure proxy: `/api/*` → `http://localhost:3000/api/*`
6. Configure WebSocket proxy: `/ws/*` → `ws://localhost:3000/ws/*`
7. Add file watching and hot-restart for backend changes
8. Support serving static files from `web/` directory

**Success Criteria:**
- `virust dev` runs both servers
- Frontend can call backend APIs
- WebSocket communication works
- Hot reload works on both frontend and backend

---

### Phase 5: Persistence Layer

**Goal:** Implement trait-based persistence with in-memory default.

**Tasks:**

1. Define `Persistence` trait in virust-protocol
2. Implement `InMemoryPersistence` using HashMap
3. Add `serialize`/`deserialize` for persistence types
4. Create `AppState` to hold persistence instance
5. Add support for dependency injection via `State(persistence)`
6. Write persistence tests and examples

**Success Criteria:**
- Trait defines clear interface for CRUD operations
- In-memory implementation works
- Easy to swap implementations
- Data persists across requests

---

### Phase 6: Working Templates

**Goal:** Build Chat and Todo templates with real functionality.

**Chat Template:**
- Real-time messaging over WebSocket
- Username handling
- Message history (persisted)
- Clean, modern UI
- Supports multiple users in same room

**Todo Template:**
- Full CRUD operations
- Dynamic routes for individual todos (/api/todos/:id)
- Mark complete/incomplete functionality
- Clean, functional UI
- RESTful API design

**Success Criteria:**
- Templates install with `virust init chat-app` and `virust init todo-app`
- Both apps run with `virust dev`
- Data persists in memory
- Frontend communicates with backend
- Real-time features work

## Persistence Layer

### Trait Definition

```rust
// virust-protocol/src/persistence.rs

pub trait Persistence: Send + Sync {
    async fn get<T>(&self, id: &str) -> Result<Option<T>, PersistenceError>;
    async fn list<T>(&self) -> Result<Vec<T>, PersistenceError>;
    async fn create<T>(&self, item: T) -> Result<(), PersistenceError>;
    async fn update<T>(&self, id: &str, item: T) -> Result<(), PersistenceError>;
    async fn delete(&self, id: &str) -> Result<bool, PersistenceError>;
}

pub struct PersistenceError {
    pub kind: ErrorKind,
    pub message: String,
}
```

### In-Memory Implementation

```rust
// virust-runtime/src/persistence.rs

pub struct InMemoryPersistence {
    data: Arc<RwLock<HashMap<String, serde_json::Value>>>,
}

impl Persistence for InMemoryPersistence {
    async fn get<T>(&self, id: &str) -> Result<Option<T>, PersistenceError> {
        let data = self.data.read().await;
        match data.get(id) {
            Some(value) => serde_json::from_value(value.clone()).map_err(|e| PersistenceError {
                kind: ErrorKind::Deserialization,
                message: e.to_string(),
            }),
            None => Ok(None),
        }
    }

    async fn list<T>(&self) -> Result<Vec<T>, PersistenceError> {
        let data = self.data.read().await;
        data.values()
            .filter_map(|v| serde_json::from_value(v.clone()).ok())
            .collect()
    }

    async fn create<T>(&self, item: T) -> Result<(), PersistenceError> {
        let id = Uuid::new_v4().to_string();
        let value = serde_json::to_value(item)?;
        let mut data = self.data.write().await;
        data.insert(id, value);
        Ok(())
    }

    // ... other methods
}
```

## What Changed from v0.1

1. **Filesystem routing** - From manual `src/api/` to true `api/` directory discovery
2. **Handler syntax** - From `#[get] pub async fn` to `pub async function GET()`
3. **TypeScript generation** - From build-time to runtime endpoint
4. **Full-stack dev** - Added Vite integration and proxying
5. **Persistence** - Added trait-based pluggable storage layer
6. **Templates** - Complete working applications instead of stubs
