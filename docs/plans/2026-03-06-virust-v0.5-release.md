# Virust v0.5 Release Notes

**Release Date:** March 6, 2026

---

## Overview

Virust v0.5 represents a **major leap forward** in the framework's capabilities, introducing **Static Site Generation (SSG)**, **Incremental Static Regeneration (ISR)**, advanced **HTTP caching**, and significant **performance improvements**. This release delivers the best of all worlds: lightning-fast static pages, fresh content with ISR, and flexible SSR when you need it.

v0.5 brings production-grade static site generation to Virust, enabling developers to build hybrid applications that mix static content, incrementally regenerated pages, and dynamic server-rendered components - all in a single codebase with zero configuration.

### What Makes v0.5 Special

- **Zero-Configuration SSG**: Just add `#[ssg]` and build static pages
- **ISR Support**: Update static pages without full rebuilds with `revalidate`
- **HTTP Caching**: Add cache control with `#[cache]` attribute
- **2.06x Performance Boost**: Parallel rendering optimizations
- **158 Passing Tests**: Comprehensive test coverage
- **Backward Compatible**: All v0.4 projects continue to work without changes
- **Production Ready**: 3 templates updated with SSG examples

## What's New

### Static Site Generation (SSG)

The headline feature of v0.5 is full static site generation support.

**Key Features:**
- Route discovery for SSG-enabled routes
- Parallel page building using all CPU cores
- Automatic HTML generation with ISR metadata
- Zero-configuration build system
- Deploy to any static hosting platform

**Usage:**
```rust
// api/route.rs
use virust_macros::{get, ssg};
use virust_runtime::RenderedHtml;

#[get]
#[ssg]
pub async fn index() -> RenderedHtml {
    RenderedHtml::new("HomePage")
}

#[get]
#[ssg]
pub async fn about() -> RenderedHtml {
    RenderedHtml::new("AboutPage")
}
```

**Build Command:**
```bash
# Build static site (outputs to dist/)
virust build --ssg

# Custom output directory
virust build --ssg --output out/

# Control parallelism
virust build --ssg --jobs 8
```

**Build Output:**
```

📂 Discovering SSG routes...
   Found 5 SSG route(s)
🔨 Building pages (8 parallel jobs)...
✅ Build complete!
   Pages built: 5
   Pages failed: 0
   Routes with ISR: 3
   Build time: 1.2s
   Output: dist/
```

### Incremental Static Regeneration (ISR)

ISR gives you the best of both worlds: static page speed with fresh content.

**Key Features:**
- Configure revalidation intervals per route
- Background regeneration after expiry
- Stale-while-revalidate serving
- ISR metadata tracking
- Zero stale page serving

**Usage:**
```rust
// api/blog/[slug]/route.rs
#[get]
#[ssg(revalidate = 3600)]  // Revalidate every hour
pub async fn blog_post(#[path] slug: String) -> RenderedHtml {
    RenderedHtml::new("BlogPost")
}

// api/products/[id]/route.rs
#[get]
#[ssg(revalidate = 300)]  // Revalidate every 5 minutes
pub async fn product_page(#[path] id: String) -> RenderedHtml {
    RenderedHtml::new("ProductPage")
}
```

**How ISR Works:**
1. **Build time** - Page pre-rendered to static HTML
2. **Request time** - Static HTML served instantly
3. **After revalidate seconds** - Background regeneration triggered
4. **Next request** - Fresh static HTML served

**Revalidation Guidelines:**

| Content Type | Revalidate Time | Example |
|--------------|----------------|---------|
| Real-time data | 60-300 seconds | Stock prices, scores |
| News articles | 300-3600 seconds | News feed, announcements |
| Blog posts | 3600-86400 seconds | Blog, documentation |
| Marketing pages | None (static) | About, pricing, contact |

### HTTP Caching Middleware

Add HTTP caching to any API route with the `#[cache]` attribute.

**Key Features:**
- Cache-Control header generation
- ETag support for cache validation
- Configurable max-age per route
- Automatic cache hit/miss tracking
- Works with SSR, SSG, and API routes

**Usage:**
```rust
use virust_macros::{get, cache};
use serde::Serialize;

#[derive(Serialize)]
pub struct ApiResponse {
    message: String,
}

#[get]
#[cache(max_age = 300)]  // Cache for 5 minutes
pub async fn get_data() -> Json<ApiResponse> {
    Json(ApiResponse {
        message: "Cached data".to_string(),
    })
}

#[get]
#[cache(max_age = 3600)]  // Cache for 1 hour
pub async fn get_config() -> Json<Config> {
    // Static configuration data
    Json(get_config())
}
```

**Cache Headers:**
```
Cache-Control: public, max-age=300
ETag: "abc123"
X-Cache: HIT
```

**Combining Caching with ISR:**
```rust
#[get]
#[ssg(revalidate = 3600)]  // ISR: regenerate every hour
#[cache(max_age = 300)]    // HTTP cache: 5 minutes
pub async fn blog_post(#[path] slug: String) -> RenderedHtml {
    RenderedHtml::new("BlogPost")
}
```

This gives you:
- **5-minute browser cache** - Reduces repeated requests
- **1-hour ISR** - Background regeneration
- **Instant static serving** - Pre-rendered HTML

### Hybrid Rendering Strategies

Mix SSG, ISR, SSR, and API routes in one application.

**Example:**
```rust
// Static pages (build once, serve forever)
#[get]
#[ssg]
pub async fn about() -> RenderedHtml {
    RenderedHtml::new("AboutPage")
}

// ISR pages (build + regenerate)
#[get]
#[ssg(revalidate = 3600)]
pub async fn blog_post(#[path] slug: String) -> RenderedHtml {
    RenderedHtml::new("BlogPost")
}

// SSR pages (render on every request)
#[get]
#[render_component("Dashboard")]
pub async fn dashboard() -> RenderedHtml {
    RenderedHtml::new("Dashboard")
}

// API routes (no rendering)
#[get]
#[cache(max_age = 300)]
pub async fn api_data() -> Json<Data> {
    // ...
}
```

```

└── virust-cli/
    ├── build.rs          # Enhanced build command
    └── main.rs           # CLI entry point
```

### SSG Build System

**Route Discovery:**
- Scans `api/` directory for `route.rs` files
- Parses `#[ssg]` attributes
- Extracts revalidate times
- Builds route metadata

**Parallel Building:**
- Uses `tokio::task::JoinSet` for parallel execution
- Semaphore-based concurrency control
- Configurable parallel jobs
- Automatic error handling and reporting

**ISR Metadata:**
- Generated during build
- Stored as `.isr-metadata.json`
- Tracks generation times
- Supports tag-based revalidation

### Caching Middleware

**CacheMiddleware:**
- In-memory cache store
- ETag generation
- Cache-Control headers
- X-Cache hit/miss tracking

**Cache Config:**
```rust
pub struct CacheConfig {
    pub max_age: Option<u64>,
    pub stale_while_revalidate: Option<u64>,
    pub tags: Vec<String>,
}
```

### Adding SSG to an Existing Project

1. **Add `#[ssg]` to Routes:**
   ```rust
   // api/route.rs
   use virust_macros::{get, ssg};

   #[get]
   #[ssg]  // Add this attribute
   pub async fn index() -> RenderedHtml {
       RenderedHtml::new("HomePage")
   }
   ```

2. **Build Static Site:**
   ```bash
   virust build --ssg
   ```

3. **Serve Static Files:**
   ```bash
   virust serve --static dist/
   ```

That's it! Your app now has static pages.

### Adding ISR to Static Routes

1. **Add Revalidation Time:**
   ```rust
   #[get]
   #[ssg(revalidate = 3600)]  // Revalidate every hour
   pub async fn blog_post(#[path] slug: String) -> RenderedHtml {
       RenderedHtml::new("BlogPost")
   }
   ```

2. **Build and Run Server:**
   ```bash
   virust build --ssg
   virust run
   ```

3. **Automatic Revalidation:**
   - Pages regenerated in background
   - Fresh content served automatically
   - No manual intervention needed

### Adding Caching to API Routes

1. **Add `#[cache]` Attribute:**
   ```rust
   #[get]

   }
   ```

2. **That's It!**
   - Cache headers automatically added
   - ETag support included
   - Browser and CDN caching enabled

## Known Limitations

### Current Limitations

1. **ISR Revalidation**: Background regeneration not yet implemented
   - Current: Stale pages served until next build
   - Planned: Automatic background regeneration
   - Workaround: Run `virust build --ssg` periodically

2. **Cache Storage**: In-memory cache only
   - Current: Cache lost on restart
   - Planned: Persistent cache options (Redis, Memcached)
   - Workaround: Use external CDN caching

3. **Build-Time Data Fetching**: Dynamic routes require manual configuration
   - Current: Must specify all dynamic route values
   - Planned: Automatic route generation from data sources
   - Workaround: Generate routes in build script

4. **Streaming SSR**: Not yet implemented
   - Current: Full HTML rendered before response
   - Planned: Streaming HTML chunks
   - Benefit: Faster Time to First Byte (TTFB)

### Planned Improvements

These will be addressed in future releases:

- **Background ISR Regeneration** (v0.6)
- **Persistent Cache Storage** (v0.6)
- **Streaming SSR** (v0.6)
- **On-Demand Revalidation** (v0.6)
- **Tag-Based Cache Invalidation** (v0.6)
- **Edge Function Support** (v0.7)
